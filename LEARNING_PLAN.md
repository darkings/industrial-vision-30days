### 第 1 节：读取 Day08 真实图像并分析

- 内容：读取 1000 us PNG、灰度统计、过曝比例。
- 代码：`test01_camera_image_analysis.py`。
- 输出：二值图、轮廓图、灰度直方图。
- 验收：已完成，阈值 100 得到 2 个有效轮廓。

### 第 2 节：多阈值对比

- 内容：阈值 60、80、100、120、140 对比。
- 代码：循环生成多张二值图和轮廓图。
- 输出：阈值对比表、轮廓数量表。
- 验收：能根据图像和数据说明阈值变化影响。
- 学习方法：
  1. 先复习 Day03 固定阈值：阈值越低，白色区域越多；阈值越高，白色区域越少。
  2. 再结合当前真实相机图像说明：当前图像不是理想黑白图，背景、反光、边缘过渡都会影响二值化。

  4. 先看示范代码结构，不急着写完整答案：阈值列表、循环、输出文件名、统计打印。

  6. 检查时同时看代码、打印数据和输出图片，不只看程序是否运行成功。
  7. 问答重点：为什么阈值低会误检背景，为什么阈值高会漏掉目标，为什么不能只凭 `len(contours)` 判断目标数量。

### 第 3 节：OTSU 接入真实相机图像

- 内容：固定阈值与 OTSU 对比。
- 代码：计算 OTSU 阈值并保存结果。
- 输出：OTSU 二值图、固定阈值对比图。
- 验收：能判断 OTSU 是否适合当前图像。
- 学习方法：
  1. 先复习 Day03 OTSU：它适合前景和背景灰度分布差异明显的场景。
  2. 再看当前相机图像直方图，判断是否存在明显双峰。
  3. 讲 API：`ret, binary = cv2.threshold(gray, 0, 255, cv2.THRESH_BINARY + cv2.THRESH_OTSU)`，重点解释 `ret` 是 OpenCV 自动计算出的阈值。
  4. 学员练习时必须同时保存固定阈值结果和 OTSU 结果。
  5. 验收时不问“哪个更高级”，只问“哪个更适合当前图像，为什么”。

### 第 4 节：形态学清理真实图像

- 内容：开运算、闭运算、面积过滤。
- 代码：清理阈值图中的小区域。

- 验收：能说明清理会影响小缺陷。
- 学习方法：
  1. 先复习 Day03：开运算先腐蚀后膨胀，闭运算先膨胀后腐蚀。
  2. 再结合真实图像观察：哪些白点是噪声，哪些边缘可能是目标细节。
  3. 讲 API：`cv2.morphologyEx()` 的输入、操作类型、kernel、iterations。
  4. 学员练习时保存三张图：原二值图、开运算后、闭运算后。
  5. 检查时重点看是否把缺陷也清理掉了，而不是只追求图像“干净”。

### 第 5 节：把相机抓帧函数接入分析流程

- 内容：从 SDK 直接拿 `numpy.ndarray`，不经过 PNG。
- 代码：`grab -> analyze -> save report`。
- 输出：实时抓帧分析结果。
- 验收：能在资源释放正确的前提下完成一次分析。
- 学习方法：
  1. 先明确边界：这节只做“一次抓帧 + 一次分析”，不做实时循环检测。
  2. 复习 Day08 SDK 生命周期：枚举、打开、开始取流、抓帧、停止取流、关闭、销毁。
  3. 复习 Day09 第 1 节图像分析函数：输入必须是 `numpy.ndarray`。
  4. 练习时先保证抓图成功，再把图传给分析函数，不要一开始就混在一起调。

```text
1. 先固定相机、镜头、工件、光源，再谈算法参数。
2. 光源控制器先学习常亮、亮度调节和基本接线，不急着做触发联动。
3. 所有曝光、阈值和轮廓结论都要基于固定后的新图像重新判断。
4. Day10 仍然不做正式尺寸标定和最终 OK/NG 结论。
```

### 第 1 节：硬件安装与安全检查

- 内容：支架固定、相机安装、镜头锁紧、光源安装、控制器供电与接线检查。
- 代码：暂不写新算法，只记录硬件状态和拍摄距离。
- 输出：硬件安装检查表、现场照片或文字记录。
- 验收：相机、镜头、光源、控制器连接稳定，无明显松动和供电异常。
- 学习方法：
  1. 先讲工业问题：算法不稳定，很多时候不是代码错，而是相机、光源、工件位置不稳定。
  2. 认识硬件角色：支架负责固定，镜头负责成像范围，环形光负责突出表面/边缘，控制器负责供电和亮度控制。
  3. 学员记录设备信息：相机型号、镜头焦距、光源类型、控制器型号、连接方式。
  4. 不写检测算法，只做安装检查和风险记录。
  5. 验收重点：是否知道每个硬件变化会影响哪些图像特征。

### 第 2 节：固定画面和 FOV 检查

- 内容：工作距离、视野范围、工件位置、画面边界。
- 代码：连续抓 5 张图并保存。
- 输出：固定后连续图像。
- 验收：主体位置不明显漂移，ROI 有足够余量。
- 学习方法：
  1. 先复习 Day07 FOV 和像素精度：视野越大，单像素代表的实际尺寸越大。
  2. 讲为什么画面里要留边距：工件轻微偏移时不能被裁掉。
  3. 代码练习只做连续抓 5 张图，保存到 Day10 outputs。
  4. 检查时对比 5 张图的主体位置，不只看有没有保存成功。
  5. 问答重点：为什么固定画面后才重新选择阈值。

### 第 3 节：环形光源安装与角度观察

- 内容：光源角度、亮度、反光、阴影、背景对比度。
- 代码：同一曝光下，比较环境光、环形光低亮度、环形光中亮度、环形光高亮度。
- 输出：光源对比图和亮度统计表。
- 验收：能判断环形光是否让目标边缘、表面纹理或反光更稳定。
- 学习方法：
  1. 先讲光源目标：不是越亮越好，而是让目标和背景更容易分开。
  2. 复习 Day07 光源知识：环形光常用于一般表面照明，但可能带来反光。
  3. 学员观察图像：边缘是否更清楚，阴影是否减少，是否出现过曝或强反光。
  4. 练习时记录每张图的光源亮度档位和曝光值。
  5. 验收时用图像和统计数据一起说明，不靠肉眼一句“更亮了”。

### 第 4 节：光源控制器常亮模式测试

- 内容：控制器供电、亮度旋钮或亮度档位、常亮输出、线缆连接。
- 代码：不同控制器亮度下抓图并统计 mean、max、overexposed_ratio。
- 输出：控制器亮度对比表。
- 验收：能说明控制器改变的是光源输出，从而改变图像灰度分布。
- 学习方法：
  1. 先认识控制器端口和接线，不做复杂触发。
  2. 讲清常亮和频闪的区别：常亮便于入门调试，频闪更偏现场高速触发。
  3. 学员先手动调亮度，再抓图记录。
  4. 检查重点：亮度变化是否导致过曝，是否影响阈值分割。

### 第 5 节：固定光源后的曝光重新扫描

- 内容：重新测试 800、1000、1200、1500 us 等曝光值。
- 代码：沿用 Day08 曝光对比脚本，保存新一轮图像。
- 输出：固定支架和光源后的曝光统计表。
- 验收：选择不过曝且亮度足够的默认曝光。
- 学习方法：
  1. 先说明为什么旧曝光参数不能直接沿用：光源和距离变了，灰度分布也会变。
  2. 复习 Day08 曝光实验：mean、max、overexposed_ratio 的意义。
  3. 学员用固定光源重新扫描曝光。
  4. 每张输出图文件名必须包含曝光值和光源条件。
  5. 验收时选“当前默认曝光”，并说明为什么不是越亮越好。

### 第 6 节：固定成像系统连续抓图稳定性

- 内容：连续抓 20 帧，统计 mean、max、轮廓数和目标框位置。
- 代码：生成稳定性 CSV 或 Markdown 表。
- 输出：固定支架和光源后的稳定性报告。
- 验收：能判断硬件固定和光源是否显著改善图像稳定性。
- 学习方法：
  1. 先讲工业问题：单张图好看不代表系统稳定，连续抓图才有意义。
  2. 统计至少 20 帧的 mean、max、过曝比例、有效轮廓数量、最大轮廓框位置。
  3. 保存 CSV 或 Markdown 表，不能只保存最后一张图。
  4. 检查时看波动范围：亮度是否漂，轮廓数量是否乱跳，目标框位置是否稳定。
  5. 验收后才能进入 Day11 的真实相机检测小流水线。

## 四、怎么学习

每天按照固定节奏学习，不要只看代码，也不要只看理论。

### 1. 每天开始前

```text
1. 打开 AGENTS.md，确认当前文档更新规则和协作方式。
2. 打开 LEARNING_PLAN.md，确认当前学习路线。
3. 打开当天 Obsidian 笔记，继续当天概念、代码和实验记录。
4. 必要时查看对应日期目录中的练习代码和 outputs。
```

### 2. 每一节课的学习顺序

```text
1. 先说明本节位置：属于哪一天、第几节、承接前面哪一节。
2. 说明工业问题：这节解决现场什么问题，为什么现在学它。
3. 讲核心概念：只讲本节需要的概念，不提前塞太多。
4. 讲 API / SDK / Python 语法：函数作用、关键参数、返回值、常见坑。
5. 先说文件名和编码目标：本节代码写在哪个文件，运行后应该得到什么结果。
6. 看示范代码或代码结构：先理解流程，再动手写。
7. 学员自己跟写：助手检查时先指出问题，不直接替学员完成答案。
8. 修改参数观察变化：每次只改 1-3 个关键参数，避免变量太多。
9. 检查输出证据：原图、中间图、结果图、打印数据、保存路径。
10. 理论问答：用自己的话解释本节结果，而不是背 API。
11. 检查 Obsidian 笔记：知识、API、代码、Markdown、工业风险都要看。
12. 记录错误：有复盘价值的概念、API、工程逻辑错误统一写入 Obsidian 的 _LearningErrors.md；当天课程笔记只保留必要易错点和关键结论。
13. 当天全部课程结束后，再总结当天核心记忆点。
```

### 3. 每天结束前

```text
1. 至少保存一张原图、一张中间结果图、一张最终结果图。
2. 记录关键参数：曝光、增益、阈值、面积阈值、光源条件。
3. 记录今天最容易错的 1-3 个点。
4. 将实验结果、路径、关键参数和结论补进当天 Obsidian 笔记。
5. README.md 和 LEARNING_PLAN.md 不再逐课更新；错误统一写入 Obsidian 的 _LearningErrors.md；路线、环境或项目规则变化时按 AGENTS.md 更新项目级文档。
```
